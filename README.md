# SportPulse API

> Production-grade REST API powering SportPulse — a sports news platform where
> articles are automatically generated by AI using real match data and sourced
> press content.

## Architecture

The codebase is organized in strict layers, each with a single responsibility:

```
config      → Environment validation, DB, Redis connections
db/schema   → Drizzle ORM table definitions, enums, migrations
lib         → Shared utilities (errors, pagination, response, retry, logger)
middleware  → Cross-cutting concerns (auth, error, rate-limit, security, CORS, logging, timeout)
modules     → Feature modules (sports, events, articles, users, teams)
            → Each module: routes → controller → service → repository
integrations→ External API clients (Mistral AI, API-Football, RSS, Guardian)
jobs        → BullMQ queues, workers, cron schedulers
docs        → OpenAPI 3.1 specification
app.ts      → Server entry point
```

### Authentication

Write operations require an `x-user-id` header containing a valid UUID:

```
x-user-id: 89d9f14d-1414-4c8d-93ce-8e81c0ca3fe2
```

This is a lightweight V1 auth mechanism — JWT will replace it in V2.

### Pagination

All list endpoints use cursor-based pagination (no offset):

**Request:**
```
GET /api/articles?limit=20&cursor=eyJpZCI6IjEyMyJ9
```

**Response:**
```json
{
  "success": true,
  "data": {
    "data": [...],
    "meta": {
      "nextCursor": "eyJpZCI6IjQ1NiJ9",
      "hasMore": true,
      "count": 20
    }
  }
}
```

Pass `meta.nextCursor` as the `cursor` param in the next request.

## Article Generation Pipeline

Every hour, a cron job fetches recently finished football fixtures from the
API-Football (via RapidAPI) and upserts teams and events into the database.
Five minutes later, a second cron identifies finished events that have no
article yet and creates a pending article record for each, then enqueues a
BullMQ job (up to 10 per run to respect API rate limits).

The worker — running with concurrency 3 — picks up each job, searches for
relevant press excerpts via RSS feeds and The Guardian API, builds a structured
prompt from the match data and excerpts, calls Mistral AI (mistral-small-latest),
parses and validates the JSON response, then saves the finished article to
PostgreSQL with status `published`. Failed jobs are retried up to 3 times with
exponential backoff (5s → 10s → 20s); after the final failure the article is
marked `failed`.
